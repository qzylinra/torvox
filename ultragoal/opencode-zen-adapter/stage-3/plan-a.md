# Plan A: Build adapter

## Approach overview

A `setup.js` (Node.js, zero external deps) that discovers free OpenCode Zen models via two API fetches, then:
1. Writes `~/.openclaude.json` with an `agentModels.zen-free` entry (base_url, api_key, model) and `agentRouting.default: "zen-free"` ŌĆö no env vars, no `.env` file, no `OPENCODE_API_KEY`.
2. Generates a runner script `opencode-zen-free` (~/.local/bin or next to setup.js) that shells out with `CLAUDE_CODE_USE_OPENAI=1` set inline ŌĆö the only env var in the whole system.

The config stores all provider-specific data; the runner only supplies the switch flag.

## How to discover free models

### Two API calls in parallel

```
GET https://models.dev/api.json
  ŌåÆ response.opencode.models: Record<modelId, { status?, cost?: { input?, output?, ... } }>

GET https://opencode.ai/zen/v1/models
  Headers: { Authorization: "Bearer public" }
  ŌåÆ response.data: Array<{ id: string }>
```

The second call uses `Authorization: Bearer public` (the free-tier key). OpenCode Zen returns the set of models visible to that key. No `OPENCODE_API_KEY` env var or config file is consulted.

### Filtering logic

```
visibleIds = new Set(response2.data.map(m => m.id))

for each (modelId, info) in modelsDevInfo:
  if visibleIds.has(modelId)
  AND info.status !== "deprecated"
  AND (info.cost?.input ?? -1) === 0          ŌåÉ exact zero check
  ŌåÆ candidate

rank candidates by info.cost?.contextWindow desc (largest context first)
pick the first as the default model
```

### Fallbacks

| Failure | Fallback |
|---------|----------|
| `models.dev/api.json` unreachable | Use a built-in hardcoded list of known-free model IDs (from pi-opencode-zen's `allModels` where cost.input === 0). |
| `/zen/v1/models` unreachable | Assume all models from models.dev are visible ŌĆö skip the intersection, just use models.dev data. Filter to free only. |
| Both unreachable | Abort with a clear error message mentioning both endpoints and suggesting `--offline` mode. |
| No free models found after filtering | Print the list of visible-but-not-free models to help debug; suggest checking key. |
| `~/.openclaude.json` unwritable | Print the path and instructions to write it manually. |

## How to store config

### `~/.openclaude.json` (written by setup.js)

The config stores everything the provider needs except the `CLAUDE_CODE_USE_OPENAI` switch:

```json
{
  "agentModels": {
    "zen-free": {
      "model": "big-pickle",
      "base_url": "https://opencode.ai/zen/v1",
      "api_key": "public"
    }
  },
  "agentRouting": {
    "default": "zen-free"
  }
}
```

- **`agentModels.zen-free`**: Defines a named agent config with the free model, the OpenCode Zen base URL, and `public` as the API key (OpenCode Zen free-tier auth).
- **`agentRouting.default: "zen-free"`**: Routes the default agent (the main session) to use `agentModels.zen-free` as its provider config. This is how openclaude discovers base_url and api_key for the main loop without env vars.
- **Merge behavior**: If `~/.openclaude.json` already exists, read it, merge the two keys (`agentModels.zen-free` and `agentRouting.default`), preserving all existing keys. Do NOT overwrite other `agentModels` entries. Do NOT overwrite the entire file.

The API key is `"public"` (not a real secret) ŌĆö this is the OpenCode Zen free tier auth token. No real credentials are stored.

### Runner script (`opencode-zen-free` ŌĆö generated by setup.js)

The only env var needed is the provider-mode switch. The runner wraps it inline:

```bash
#!/usr/bin/env bash
# Generated by setup.js ŌĆö run this instead of `openclaude` to use free models
# Install: chmod +x ~/.local/bin/opencode-zen-free
exec env CLAUDE_CODE_USE_OPENAI=1 openclaude "$@"
```

- `exec env CLAUDE_CODE_USE_OPENAI=1 openclaude "$@"` sets the var only for the child process ŌĆö no `export`, no shell pollution.
- The runner is installed to `~/.local/bin/opencode-zen-free` (or the same directory as setup.js). `~/.local/bin` is a standard PATH entry on most systems.

---

## The runner script design

Design goals:
- Only `CLAUDE_CODE_USE_OPENAI=1` as the env var (the switch to OpenAI-compatible mode)
- No `OPENAI_BASE_URL` / `OPENAI_API_KEY` / `OPENAI_MODEL` in the runner ŌĆö those come from `agentModels`
- No `.env` file created or referenced
- No `--provider-env-file` flag used

The runner uses `exec env` to pass the single env var without modifying the parent shell. This is the **only** env var anywhere in the system.

## Error handling summary

| Scenario | Behavior |
|----------|----------|
| Node.js < 22 | Print error: "Node.js >= 22 required" |
| openclaude not found | Warning but continue (user may add to PATH later) |
| Both API endpoints down | Abort with error message showing both URLs |
| Only models.dev down | Use built-in model list, continue |
| Only /zen/v1/models down | Use models.dev data (no intersection) |
| No free models found | Print available (paid) model IDs, suggest getting an API key |
| ~/.openclaude.json unreadable | Start from empty config |
| ~/.openclaude.json unwritable | Print content to stdout with copy instructions |
| Runner output path unwritable | Print runner content with copy instructions |
