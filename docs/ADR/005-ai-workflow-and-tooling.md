# ADR 005: AI 工作流与工具链 — 规范驱动开发 + 多智能体协作

**状态**: 已接受
**日期**: 2026-05-26
**决策者**: 项目负责人

---

## 上下文

Torvox 是 AI 辅助开发的绿地项目。从 100+ AI 重写案例中提取的核心教训是：**规范先行 > vibe coding 10 倍**。本项目需要定义 AI 开发工作流、工具配置和质量标准。

## 决策

**采用规范驱动开发 (SDD) 工作流，配置多 AI 工具协作，严格分阶段门控。**

### 规范驱动开发流程

```
1. 编写精确规范 (ADR/SPEC) ← 人类做架构决策
2. 定义类型和接口 ← 先类型后行为
3. AI 生成实现 ← 基于规范的代码生成
4. 人类审查 + 测试 ← 每个生成的函数
5. cargo clippy + test ← 自动化质量门
6. 提交 ← Conventional Commits
```

这与 Bun 1M 行/9 天重写的方法论一致：先写完整规范, 再让 AI 执行实现。

### AI 工具配置

| 工具 | 角色 | 配置位置 |
|------|------|----------|
| **opencode** | 主要 AI 编码工具 | `AGENTS.md` (仓库根), `opencode.jsonc` |

### 配置文件层次

```
torvox/
├── AGENTS.md              # 主 AI 上下文文件 (opencode 兼容)
├── opencode.jsonc         # opencode 配置
└── .opencode/agents/      # 自定义智能体定义
```

### AGENTS.md 结构 (基于最佳实践)

```markdown
# 项目标识
# 关键文件
# 当前状态
# 关键约束 (禁止/必须)
# 构建命令
# 约定 (Rust/Kotlin/Git)
# 测试要求
# 架构提醒
# 会话结束检查清单
```

### opencode 配置

见 `opencode.jsonc`。定义 3 个专用智能体 (torvox-rust, torvox-kotlin, torvox-docs) + 质量门命令。

### AI 交互的 7 种提示模式

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **规范型** | 提供精确规范让 AI 实现 | 新 crate/模块实现 |
| **示例型** | 提供参考代码让 AI 模仿 | 遵循现有模式 |
| **渐进型** | 分步骤引导 AI 实现 | 复杂多步任务 |
| **约束型** | 明确禁止/必须的约束 | 安全关键代码 (PTY, unsafe) |
| **分解型** | 将大任务分解为小任务 | 阶段里程碑 |
| **角色型** | 指定 AI 角色 (审查者/实现者) | 代码审查 |
| **对话型** | 探索性讨论 | 架构决策 |

### AI 交互的 4 种反模式 (严格禁止)

| 反模式 | 描述 | 失败案例 |
|--------|------|----------|
| **无规范编码** | 让 AI 在无规范的情况下生成代码 | syntaqlite 原型被废弃 |
| **忽略测试** | AI 生成代码但不写测试 | 社区报告: 60% AI 代码有 bug |
| **批量修改** | 一次让 AI 修改 10+ 文件 | 引入跨文件不一致 |
| **抽象泄漏** | AI 生成代码引用不存在的模块 | 编译失败, 幻觉 API |

### 分阶段门控

每个阶段有明确退出标准，**未通过退出标准不得进入下一阶段**：

```
阶段 N 开始
→ 编写阶段规范 (ADR 更新)
→ AI 生成实现
→ 人类审查
→ cargo clippy + nextest + geiger
→ 退出标准检查
→ [通过] → 阶段 N+1
→ [未通过] → 修复 + 重试
```

这与 Google Ads 500M 行重写的方法论一致。

### TDFlow 四智能体循环 (测试策略)

```
测试智能体 → 编写失败测试
↓
调试智能体 → 分析失败原因
↓
修复智能体 → 实现修复
↓
验证智能体 → 确认修复 + 回归测试
↓
[通过] → 完成
[未通过] → 回到调试智能体
```

这来自 TDFlow 论文 (SWE-Bench 88.8%)。

## 后果

**正面**:
- AI 辅助开发效率 10x (Bun 案例: 1M 行/9 天)
- 分阶段门控防止大规模错误
- 规范先行确保架构一致性
- 多工具协作覆盖不同场景

**负面**:
- 规范编写时间成本 (估计 30% 开发时间用于规范)
- AI 工具链维护成本 (配置文件同步)
- 人类审查疲劳 (需要每个 AI 生成的函数都审查)

**缓解措施**:
- `cargo clippy --deny warnings` + `cargo nextest` 自动化质量门
- `cargo geiger` + `MIRI` 自动化安全检查
- ADR 模板化减少规范编写时间
- `proptest` 属性测试自动发现 AI 代码中的 bug
